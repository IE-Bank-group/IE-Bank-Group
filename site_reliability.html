Service Level Agreement (SLA)

Effective Date: 10/08/2024
Version: 2.0

Introduction

This Service Level Agreement (SLA) outlines the service commitments provided by Ie Bank regarding the availability, performance, and security of its banking application hosted on Azure. The purpose of this document is to ensure that users have clarity on the service levels they can expect, and to foster trust and transparency in how Ie Bank measures, monitors, and maintains its critical banking services.

Scope of Services

This SLA applies to the following components of the Ie Bank application ecosystem:

  • Frontend: Azure Static Web App or Azure Front Door (if applicable)
  • Backend Services: Azure App Service (running containers or web APIs)
  • Database: Managed PostgreSQL Database on Azure
  • Security & Credentials Management: Azure Active Directory (Azure AD), Azure Key Vault
  • Monitoring & Logging: Azure Application Insights, Azure Monitor, Azure Log Analytics Workspace
  • Alerting & Incident Management: Integrated Slack Notifications, Azure Alerts

Note: Overall performance is dependent on these interconnected services. A degradation in one component may impact the entire system’s service quality.

Service Availability

Ie Bank commits to maintaining high availability for core services:

  • Overall System Availability: 99.7% uptime over a rolling 30-day period
  • Frontend Services (Web App): 99.7% uptime over a rolling 30-day period
  • Backend Services (API/Container Apps): 99.7% uptime over a rolling 30-day period
  • Database (PostgreSQL): 99.7% uptime over a rolling 30-day period
  • Security & Key Vault Services: 99.7% uptime over a rolling 30-day period
  • Monitoring & Logging Services: 99.7% uptime over a rolling 30-day period

Performance Metrics (SLOs)

Ie Bank defines the following Service Level Objectives (SLOs) to guide performance targets:

  • Availability:
    • SLI: Percentage of uptime (SuccessfulRequests/TotalRequests)
    • SLO: 99.7% availability over 30 days
    • Measurement: Track service availability via logs and uptime checks
    • Tools: Azure Monitor, Application Insights
  • Latency (API Response Time):
    • SLI: 95th percentile response time for API requests
    • SLO: 95% of requests < 300ms over a 7-day period
    • Measurement: Collect and analyze API response times from logs and performance tests
    • Tools: Azure Application Insights, Postman tests
  • Error Rate:
    • SLI: Percentage of failed API requests (4xx/5xx)
    • SLO: <0.1% error rate over a 7-day period
    • Measurement: Count failed requests via API error logs
    • Tools: Azure Monitor, Log Analytics
  • Security (Authentication Success Rate):
    • SLI: Rate of successful authentication attempts
    • SLO: 99.99% of authentication attempts succeed without breaches
    • Measurement: Monitor authentication logs for success/failure ratios
    • Tools: Azure AD, Key Vault, Security Center
  • Data Integrity (Database Transaction Success):
    • SLI: Rate of successful database transactions
    • SLO: 99.95% of transactions succeed over a 7-day period
    • Measurement: Monitor database transaction logs for success vs. failure rates
    • Tools: PostgreSQL Monitoring Tools, Azure Monitor
  • Monitoring & Alerts (Incident Response):
    • SLI: Percentage of alerts acknowledged and resolved within target times
    • SLO: 90% of critical alerts acknowledged within 5 minutes and resolved within 30 minutes
    • Measurement: Track alert response and resolution times
    • Tools: Azure Monitor, Slack Notifications

Penalties for Non-Compliance

If Ie Bank fails to meet the specified availability or performance metrics, customers may be eligible for service credits:

  • For each hour of downtime beyond the 99.7% availability threshold, a credit of up to 2% of monthly fees (up to a maximum of 30%) may be applied.
  • If a given performance SLO (e.g., latency, error rate) falls below its target for two consecutive measurement periods, Ie Bank will review the issues with the customer and may grant service credits up to 30% of monthly fees.

These credits serve as a means of accountability and incentive for Ie Bank to continuously improve its services.

Exclusions

The SLA does not apply to:

  • Scheduled maintenance windows, communicated in advance.
  • Downtime or performance issues caused by factors outside of Ie Bank’s reasonable control (e.g., upstream Azure outages, force majeure events).
  • Incidents resulting from misuse of services or non-adherence to best practices and security guidelines by the customer.

Monitoring and Reporting

Ie Bank will utilize Azure Monitor, Azure Application Insights, and Azure Log Analytics Workspace for continuous real-time monitoring and alerting. Key aspects:

  • Real-Time Dashboards: For immediate visibility into performance and availability.
  • Monthly Reports: Summarize uptime percentages, latency metrics, error rates, security authentication success rates, data integrity, and alert response times.
  • Incident Reviews: In case of SLA violations or performance anomalies, reviews and root cause analyses will be conducted and reports shared with stakeholders.

Review and Revision of SLA

This SLA will be reviewed quarterly or whenever significant changes occur in the service infrastructure or business requirements. Amendments will be communicated to stakeholders at least 30 days in advance of taking effect.

Service Level Objectives (SLO)

  • Availability: 99.9% uptime over a rolling 30-day period
  • Latency (API Response): 95% of requests respond in under 300ms over a 7-day period
  • Error Rate: <0.1% error rate (4xx/5xx) over a 7-day period
  • Security (Authentication): 99.99% of authentication attempts succeed without breaches
  • Data Integrity: 99.95% of database transactions succeed over a 7-day period
  • Monitoring & Alerts: 90% of critical alerts acknowledged within 5 minutes and resolved within 30 minutes

Service Level Indicators (SLI)

  • Availability (Uptime): (Successful Requests / Total Requests) ≥ 99.9%
  • Latency (Response Time): (Requests <300ms / Total Requests) ≥ 95%
  • Error Rate (Failed Requests): (Successful Requests / Total Requests) ≥ 99.9% or (Failed Requests/Total <0.1%)
  • Security (Auth Success): (Successful Auth Attempts / Total Auth Attempts) ≥ 99.99%
  • Data Integrity (DB Transactions): (Successful DB Tx / Total DB Tx) ≥ 99.95%
  • Monitoring & Alerts (Response Time): (Critical Alerts Acknowledged ≤5min and Resolved ≤30min / Total Critical Alerts) ≥ 90%

Monitoring Strategy

Goal: Ensure that critical banking services meet defined SLOs for availability, performance, security, data integrity, and responsiveness to issues.

Our monitoring strategy focuses on proactive detection, rapid incident response, data-driven optimization, and transparent reporting.

Azure Services for Monitoring

  • Azure Application Insights: End-to-end visibility into application performance, response times, error rates, and user interactions.
  • Azure Log Analytics Workspace: Centralized logging and analysis for system/application logs, security events, and infrastructure metrics.
  • Azure Monitor (Metrics & Alerts): Baseline infrastructure metrics (CPU, memory) and unified alerting for quick detection of issues.
  • Azure Workbooks: Visualization of metrics, logs, and traces into interactive dashboards for SLO compliance status and root cause analysis.

Monitoring Methods

Application Performance Monitoring

Objective: Ensure frontend and backend maintain performance targets (95% requests <300ms, error rate <0.1%).

Data Needs: Page load times, API response times, error counts, authentication success rates.

Implementation:

  • Integrate Application Insights SDK into frontends and APIs.
  • Collect metrics on latency, error frequency, authentication results.
  • Configure Azure Workbooks for daily/weekly SLO trend monitoring.

Capability & Business Transaction Monitoring

Objective: Ensure key transactions (e.g., DB operations, fund transfers) meet Data Integrity (99.95%) and performance standards.

Metrics to Monitor: Successful vs. failed DB transactions, transaction times, authentication attempts, alert response times.

Implementation:

  • Use custom events in Application Insights to track transaction steps.
  • Query Log Analytics Workspace for DB logs and correlate with front-end/back-end events.
  • Set time-based queries and thresholds; alerts trigger via Azure Monitor for anomalies.

Fault Tolerance & Infrastructure Monitoring

Objective: Detect/respond to CPU spikes, memory bottlenecks impacting Availability and Latency.

Metrics: CPU usage, memory, network latency, container health.

Implementation:

  • Use Azure Monitor metrics (CPU <80%, etc.) to detect infra issues.
  • Log Analytics captures container restarts, DB performance logs.
  • Alerts to Slack for rapid incident response when thresholds exceeded.

Data Collection & Analysis

All metrics/logs flow into Azure Log Analytics Workspace and Application Insights. Diagnostic settings ensure comprehensive logging. Data is analyzed near real-time, alerts fire within minutes of threshold violations.

Visualizations & Reporting

Azure Workbooks provide dashboards for:

  • Availability, latency over time
  • Error rates and recent incidents
  • Authentication success percentages
  • Database transaction success rates
  • Alert acknowledgment and resolution times

Management Controls

  • Access restricted to authorized personnel.
  • Quarterly audits on data access and monitoring configurations.
  • SLA and SLO reviews conducted quarterly.

Incident Response Design

Version: 1.0
Effective Date: 10/08/2024

Purpose

This document outlines the incident response procedures for the Ie Bank application. Its purpose is to minimize downtime, address performance and security issues rapidly, and maintain compliance with established SLOs and SLAs.

Incident Response Objectives

  • Minimize Downtime: Maintain 99.7% uptime.
  • Ensure SLO Adherence: Resolve incidents impacting latency, error rates, authentication, data integrity, alerts.
  • Rapid Resolution: Identify and fix issues promptly.
  • Clear Communication: Keep internal teams/stakeholders informed.

Detection

  • Tools: Application Insights, Azure Log Analytics Workspace, Azure Monitor & Alerts.
  • Alerts & Thresholds:
    • Availability Alert: If successful requests drop below 99.7%.
    • Latency Alert: p95 response times > 300ms sustained.
    • Error Rate Alert: Error rate >0.1%.
    • Security Alert: Auth success <99.99%.
    • Data Integrity Alert: DB success <99.95%.
    • Monitoring & Alerts Incident: Unacknowledged critical alerts >5 min or unresolved >30 min.
    • CPU Usage Alert: CPU usage >80% for >5 min.

Incident Response Steps

Detection: Alert fires (e.g., “Error rate above 0.1%”).

Notification: Slack notification sent to on-call engineer.

Triage:

  • On-call engineer reviews Azure Workbooks dashboards.
  • Identify impacted component and severity (Critical, High, Medium, Low).

Response Actions by Severity:

  • Critical Incidents:
    • Restart/scale App Services.
    • Check Azure Resource Health.
    • If auth compromised, disable suspicious credentials.
  • High/Medium Incidents:
    • Optimize DB queries or API endpoints.
    • Adjust scaling parameters.
    • Investigate/rollback recent deployments.
  • Low Incidents:
    • Log issue, schedule future optimizations.

Resolution: Validate metrics return to normal. Confirm no lingering issues.

Post-Incident Review: Conduct retrospective, identify root causes, implement preventive measures.

External Communication

For Critical/High incidents impacting users (e.g., transaction slowdown), notify customers via email or in-app alert:

“We are experiencing slower-than-normal transaction processing. Our team is working to resolve this quickly. Thank you for your patience.”

Key Tools and Automation

  • Tools: Application Insights, Log Analytics Workspace, Azure Workbooks, Azure Monitor & Alerts.
  • Automation: Slack integration for notifications, Azure Resource Health checks, Runbooks for predefined mitigation steps.

Metrics for Measuring Incident Response

  • Time to Detect (TTD): Time from incident occurrence to alert firing.
  • Time to Acknowledge (TTA): Time from alert to on-call engineer acknowledgment.
  • Time to Resolve (TTR): Time from alert to full resolution.

Post-Incident Review Process

Root Cause Analysis: Identify underlying causes (e.g., DB query issues, code regressions).

Impact Analysis: Assess if SLOs/SLAs were breached.

Preventive Measures: Adjust thresholds, improve code, add auto-scaling rules, update documentation/runbooks.

Review, Revision, and Acceptance

This incident response plan is reviewed quarterly or after major incidents. Updates communicated to stakeholders and on-call staff.

Reliability Design

Business Requirements & Reliability Targets:

Availability Targets:

  • Overall System: 99.7% over a rolling 30-day period
  • API/Backend Services: 99.7%
  • Database (PostgreSQL): 99.7%
  • Frontend Services: 99.7%

Performance Targets:

  • Latency (API): 95% of requests <300ms (7-day period)
  • Error Rate: <0.1% of requests failing (7-day period)
  • Authentication Success: 99.99%
  • Data Integrity (DB Transactions): 99.95% success rate (7-day period)

Recovery Objectives:

RTO (Recovery Time Objective):

  • Critical Services (API, DB): ≤4 hours
  • Non-Critical Services (Batch jobs, analytics): ≤8 hours

RPO (Recovery Point Objective):

  • Transactions: ≤5 minutes
  • User Data (Profiles, Credentials): ≤1 hour
  • Analytics/Reporting Data: ≤24 hours

Resilience Strategy:

  • Component Redundancy: Multiple backend instances, load balancing, cross-region deployments.
  • Database Layer: Primary-secondary replication, point-in-time backups, geographic redundancy.
  • Fault Isolation: Circuit breakers, bulkhead patterns for resource isolation.

Recovery Design:

  • Backup Strategy: Transaction logs every 5min, full DB daily, version-controlled config backups.
  • Disaster Recovery (DR): Secondary region failover, DR runbooks, data consistency checks, DR drills.
  • Failover Process: Automated triggers, manual override, data integrity validation post-failover.

Operational Reliability:

  • Monitoring & Alerting: Track system vitals, performance metrics, error rates. Alerts with severity, Slack notifications.
  • Incident Response: Runbooks for rapid issue mitigation, internal/external communication, RCA post-incident.

Testing & Validation:

  • Reliability Testing: Unit, integration, load, stress tests to ensure correctness and performance.
  • Chaos Engineering: Failure injection, network degradation tests to strengthen resilience.
  • Recovery Testing: Backup restore drills, RTO/RPO validation, performance checks post-restore.

Continuous Improvement:

  • Performance Optimization: Query tuning, caching, auto-scaling, code optimization.
  • Capacity Planning: Forecast growth, scale resources, adjust infrastructure.
  • Process Enhancement: Update runbooks, adopt best practices, training sessions.

Compliance & Security:

  • Data Protection: Encryption at rest/in transit, access controls, audit logs, retention policies.
  • Security Measures: Authentication, authorization, vulnerability scanning, WAF.
  • Business Continuity: Impact analyses, risk mitigation strategies, crisis management, stakeholder communication.
